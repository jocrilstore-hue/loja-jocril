## What shipped

All Eupago payment + order creation surfaces ported from `SITES/loja-jocril` with email calls stripped (B7). 5 new API routes live.

| Route | Method | Purpose |
|---|---|---|
| `/api/orders` | POST | Create order via `create_complete_order` RPC (atomic, locks stock) |
| `/api/orders` | GET | Fetch by `order_number` (guest-safe) OR list for auth user |
| `/api/orders/[orderNumber]/status` | GET | Payment status (used by MB Way polling) |
| `/api/payment/multibanco` | POST | Generate MB entity+reference via Eupago, write to `orders` |
| `/api/payment/mbway` | POST | Initiate MB Way push request, write to `orders` |
| `/api/webhooks/eupago` | POST | Callback receiver — marks order paid, validates chave_api + amount tolerance 0.01 |
| `/api/webhooks/eupago` | GET | Health check |

## Files touched

### New
- `lib/payments/eupago.ts` (~220 lines after reformat) — Eupago REST wrapper. Exports `createMultibancoReference`, `createMBWayPayment`, `verifyCallback`, `formatReference`, `validatePhoneNumber`, `formatPhoneForEuPago`, `maskPhoneNumber`, `EuPagoError`, `eupagoCallbackSchema`, `EuPagoCallback`. Uses legacy REST for Multibanco, v1.02 API for MB Way.
- `app/api/payment/multibanco/route.ts` — POST handler. Idempotent: returns existing reference if already generated.
- `app/api/payment/mbway/route.ts` — POST handler. Regex `^9[1236]\d{7}$` for phone validation.
- `app/api/webhooks/eupago/route.ts` — POST + GET. Always returns 200 (even on invalid payload) to prevent Eupago retry storms. Validates `chave_api` + amount tolerance 0.01. Email notification deferred to B7.
- `app/api/orders/route.ts` — POST + GET. POST calls `create_complete_order` RPC with validated customer/shipping/items. Includes server-side price-tampering check against `product_variants.base_price_including_vat`. Email send deferred to B7.
- `app/api/orders/[orderNumber]/status/route.ts` — GET. Guest path returns only `paymentStatus`; auth path returns full status + paidAt with ownership check.

### Deps
- Installed `zod` (with `--legacy-peer-deps` again — same React 19.0.0 vs Clerk peer situation persists from B2).

### NOT ported
- `lib/email/send-order-emails` — deferred to B7. Three call sites stripped: webhook (`sendPaymentReceived`), orders POST (`sendOrderConfirmation`, `sendAdminNotification`).
- `/api/shipping/calculate` — skipped. Cart page currently uses hardcoded rates (4.90 / 7.90 / 0). B4b decides if dynamic calc needed.

## Gotchas

1. **Email stubbed out.** When B7 lands, re-wire:
   - Webhook POST after order paid → `sendPaymentReceived(orderNumber)`
   - Orders POST after RPC success → `sendOrderConfirmation(orderNumber) + sendAdminNotification(orderNumber)`
   - Both are fire-and-forget in the old project.
2. **`create_complete_order` RPC not verified in DB this session.** Old project runs against the same prod Supabase and uses it. If B4b checkout fails with `function does not exist`, copy migration from `SITES/loja-jocril/supabase/migrations/20251227200000_create_complete_order_rpc.sql` (name from memory — verify).
3. **`--legacy-peer-deps` needed for zod too?** No — zod has no React peer. Used the flag for consistency with prior install so npm doesn't complain about the pre-existing Clerk deviation.
4. **`orders.total_amount_with_vat` is the field name** (not `total`) — both payment routes query this column. Verify DB schema at checkout wire-up.
5. **Eupago webhook URL must be `${NEXT_PUBLIC_SITE_URL}/api/webhooks/eupago`**. In prod, register this in the Eupago dashboard. In dev, you need a tunnel (ngrok / cloudflared) pointing at localhost:3000.
6. **MB Way success/fail URLs hardcoded to `/checkout/sucesso`, `/checkout`, `/carrinho`.** New project has no `/checkout` page — flow uses `/carrinho` for steps 1+2, success in step 3. Update `createMBWayPayment()` URLs when B4b lands (or point `successUrl` to `/carrinho?step=3&order=...`).
