## What was completed

### API routes created (all under `app/api/admin/`)

| Route | Methods | Body / Notes |
|---|---|---|
| `/api/admin/products` | GET, POST | GET: list all product_templates + variant_count + category join. POST: `{ name, slug?, category_id?, is_featured?, is_active? }` |
| `/api/admin/products/[id]` | PATCH, DELETE | PATCH: partial product_template update. DELETE: soft delete (is_active = false) |
| `/api/admin/products/[id]/variants` | POST | `{ sku, size_format, base_price_including_vat, stock_quantity, is_active? }` |
| `/api/admin/products/[id]/variants/[variantId]` | PATCH, DELETE | PATCH: includes stock_quantity (stock management). DELETE: soft delete |
| `/api/admin/orders/[id]` | PATCH | `{ status }` — validated against enum. Sets paid_at on first paid-status transition |

All routes: auth via `getAdminContext()` from `lib/auth/admin.ts`, return 403 if not admin, zod validation, `{ success, data/error }` envelope.

### Forms wired

- **`components/admin/ProductFormPage.tsx`** — "Guardar rascunho" and "Publicar" buttons now call real API. Create → POST `/api/admin/products`, redirects to `/admin/produtos` on success. Edit → PATCH `/api/admin/products/[id]`. Error shown inline. Controlled fields: `name`, `slug`. Other tabs (Images, Content, SEO, Applications, Settings) remain visual-only — deferred to API expansion.

- **`components/admin/VariantFormPage.tsx`** — "Criar variante" / "Guardar alterações" buttons now call real API. Create → POST `/api/admin/products/[id]/variants`, redirects to `/admin/produtos/[id]` on success. Edit → PATCH `/api/admin/products/[id]/variants/[variantId]`. Controlled fields: `sku`, `size_format`, `base_price_including_vat`, `stock_quantity`, `is_active`. Added explicit stock_quantity input field.

- **`app/admin/encomendas/[id]/page.tsx`** — Added status selector dropdown + `updateStatus()` → PATCH `/api/admin/orders/[id]`. Shows success/error feedback inline. Header label and Tag update reactively.

### Schema confirmed (ground truth from migrations + lib/queries/products.ts)
- Products table: `product_templates` (NOT `products`) — `id, name, slug, is_active, is_featured, category_id, created_at, updated_at`
- Variants table: `product_variants` — FK is `product_template_id`, columns: `sku, size_format, base_price_including_vat, stock_quantity, is_active`
- Orders table: `orders` — status values: `pending | paid | prep | shipped | delivered | returned`; `paid_at TIMESTAMPTZ` confirmed (migration `20251220110000_add_order_extra_fields.sql`)
- Auth: `getAdminContext()` from `lib/auth/admin.ts` (no `permissions.ts` exists)

## Surprises / Gaps

1. **`product_templates` not `products`** — task spec said `products` table but the NEW project storefront exclusively uses `product_templates`. All admin routes adapted accordingly.
2. **`userIsAdmin()` does not exist** — task spec referenced `lib/auth/permissions.ts`. Used `getAdminContext()` from `lib/auth/admin.ts` instead.
3. **Variant FK is `product_template_id`** — not `product_id`. Confirmed from `lib/queries/products.ts` join pattern.
4. **Admin list pages (`/admin/produtos`, `/admin/encomendas`) remain mock** — they are `"use client"` components with hardcoded data. Wiring them to real Supabase queries requires converting to server components or adding a fetch-on-mount effect. Deferred — GET `/api/admin/products` and GET `/api/admin/orders` (not built, only PATCH exists for orders) can support this in B7.
5. **`size_format` in product_variants** — the column stores a string directly (not a FK to size_formats table for the admin write path). The storefront joins `size_formats` table for display but the insert only needs the string value.
6. **Tabs beyond "basic" in ProductFormPage** — not wired. The form has 6 tabs (images, content, SEO, applications, settings). Only name/slug/is_active submit to the API. Full wiring requires API expansion for those fields.
