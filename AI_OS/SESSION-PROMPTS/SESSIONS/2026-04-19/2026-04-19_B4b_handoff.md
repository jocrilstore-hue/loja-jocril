## What the checkout flow does now

1. **Step 2 — Address + Shipping + Payment** — all fields controlled state (`name`, `email`, `phone`, `nif`, `address`, `city`, `postalCode`, `country`, `mbwayPhone`)
2. **"Finalizar encomenda →"** calls `handleFinalizar()`:
   - Required field validation (name, email, phone, address, city, postalCode)
   - MB Way phone regex `^9[1236]\d{7}$`
   - POST `/api/orders` → maps cart items, customer, shipping, totals
   - Error mapping: `INSUFFICIENT_STOCK` / `PRODUCT_NOT_FOUND` / `price_changed` → PT banners
   - `clearCart()` immediately on order success (captures `orderTotal` before clear)
   - POST `/api/payment/multibanco` or `/api/payment/mbway`
   - `setOrderResult(...)` + `setStep(3)`
3. **Step 3 — Success panel** — branches on `orderResult.type`:
   - `multibanco` → `MultibancoPanel` (entity, reference, amount, deadline)
   - `mbway` → `MBWayPanel` (masked phone, polls `/api/orders/[orderNumber]/status` every 3s until `paymentStatus === 'paid'`)
4. **Error banner** — uses `--color-danger` / `--color-danger-bg` (no hardcoded hex)
5. **MB Way paid banner** — uses `--color-success` / `--color-success-bg`

## Gotchas

1. **`orderId` param in payment routes is actually `orderNumber`** — both `/api/payment/multibanco` and `/api/payment/mbway` accept `{ orderId }` but query DB by `order_number`. Pass `orderNumber` from the order creation response as `orderId`.
2. **MB Way phone entered twice** — user fills contact phone in AddressBlock AND a separate MB Way phone in PaymentOptions. UX debt: could pre-fill from contact phone, but left for B5/UX pass. Does not affect function.
3. **`create_complete_order` RPC not verified against live DB** — if checkout fails with `function does not exist`, copy migration from old project: `SITES/loja-jocril/supabase/migrations/20251227200000_create_complete_order_rpc.sql`.
4. **Eupago webhook dev testing** — needs a tunnel (cloudflared / ngrok) pointing to localhost:3000. Register `NEXT_PUBLIC_SITE_URL/api/webhooks/eupago` in Eupago sandbox dashboard.
5. **`price_changed` error has no `code` field** — the API returns `{ error: "price_changed" }` (no separate `code`). The handler checks `orderData.code || orderData.error` so it catches both.
