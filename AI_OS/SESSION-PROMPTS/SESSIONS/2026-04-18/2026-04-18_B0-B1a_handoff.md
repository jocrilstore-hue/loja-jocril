## Critical discoveries for next sessions

### The `search_products` RPC is narrow
Migration `20251220200000_improve_search_function.sql` defines the RPC to return:
```
variant_id, template_name, size_name, orientation,
category_name, base_price_including_vat, main_image_url,
url_slug, stock_status, relevance_score
```
It does **NOT** return: `sku`, `material_name`, `is_bestseller`, `template_short_description`, `template_category_id`. The adapter currently fills `sku` from `url_slug` as a placeholder and `material` as empty string. The old project's PLP code referenced fields that this RPC doesn't return ‚Äî either the old code was silently broken, or the live DB has a newer version of the function. **Verify against live DB before wiring PLP.**

### URL slugs live on variants, not templates
`product_variants.url_slug` is the canonical route key. `/produtos/[slug]` resolves to a specific variant, and the PDP then fetches sibling variants via `product_template_id`. The adapter respects this.

### RLS is legacy/bypassed
Per old project's ARCHITECTURE.md: all DB writes use service-role client (`lib/supabase/admin.ts`). RLS policies exist but Clerk user IDs don't populate `auth.uid()`. Keep using service-role for admin ops in B5/B6.
