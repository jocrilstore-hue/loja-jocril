# Forensic Debug Loop

Run a disciplined debug loop for problems that are not solved by a single inspection or patch. Favor cumulative evidence over intuition, and keep progress durable across long sessions.

This is a multi-tool skill — use it with Claude Code, ChatGPT, Gemini, Antigravity, Codex, or any other AI coding agent.

## Core Rules

- Prefer artifact-first debugging over symptom-first guessing.
- Separate the first broken stage from downstream symptoms.
- Fix one provable bug at a time, then retest from the same artifact chain.
- Turn every confirmed bug into a regression test before moving on.
- Keep a running handoff document for findings, hypotheses, fixes, and remaining risks.
- Treat live/runtime inspection as a source of truth when local tests and reality diverge.

## Start the Session

1. Define the exact failure in concrete terms.
2. Identify the artifact chain for this problem.
3. Find the highest-signal artifact already available.
4. Record the current diagnosis and open questions in a handoff doc.

Use the artifact hierarchy in `references/checklists.md` to decide what to inspect first.

## Run the Loop

1. **Reproduce** the failure consistently.
2. **Inspect** the earliest trustworthy artifact instead of jumping to the live symptom.
3. **Write a narrow diagnosis** that explains the current evidence and explicitly mark uncertainty.
4. **Patch** the smallest general rule that could fix the issue.
5. **Add or update regression tests** that lock in the expected behavior.
6. **Run the smallest relevant test set first**, then broader validation if it passes.
7. **Rebuild** any generated outputs that the user actually depends on.
8. **Re-test in the real environment.**
9. **Update the handoff doc** with what changed, what passed, and what is still unknown.
10. **Count your fix attempts.** If this is the 2nd failed patch for the same issue, stop the loop. Do not start cycle #3. Write the architectural escalation report (see below) and present it to the user.

If the real environment still fails, do not restart from scratch. Compare the new artifact to the previous one and isolate the next broken stage.

## The 2-Strikes Rule: When Fixes Keep Failing

You've tried 2+ patches and each one either reveals a new problem in a different place, requires massive refactoring, creates new symptoms elsewhere, or uncovers more shared state or coupling than expected.

This is not a bug. This is a wrong architecture.

```
IF 2+ fix attempts have failed:
  → STOP. Do not attempt fix #3.
  → The problem is structural, not local.
  → Escalate to the user for an architectural decision.
```

When escalating, present:

```
ARCHITECTURAL ESCALATION — 2+ fixes failed

Attempted fixes:
1. [Fix 1] → Failed because [reason]
2. [Fix 2] → Failed because [reason]

Pattern observed:
[What each fix revealed — coupling, shared state, wrong abstraction]

Root question:
Is [pattern/approach/architecture] fundamentally sound, or should we
reconsider the approach?

Options:
A. [Refactor the architecture — describe what changes]
B. [Accept the limitation and work around it — describe tradeoff]
C. [Scrap this approach entirely — describe alternative]
```

Continuing to patch after 2 failures is the debugging equivalent of sticking with it through sheer inertia. Each failed fix is evidence the problem is structural. Use that evidence to make a real decision.

## Keep the Investigation Cumulative

Use this pattern throughout the session:

- Hypothesis: what you think is broken
- Evidence: the exact dump, payload, DOM state, test, or runtime observation that supports it
- Fix: the smallest rule change made
- Verification: the exact tests and live checks performed
- Residual risk: what still is not proven

Do not collapse multiple unrelated failures into one narrative unless a shared cause is directly supported.

## Choose the Right Inspection Level

Escalate in this order:

1. Local code path and targeted tests
2. Integration output generated by the codebase
3. Built/generated artifact actually used by the user
4. Clipboard/export/payload dump from the real run
5. Designer/runtime inspection
6. Original-vs-converted comparison in a browser inspection tool

Escalate only when the current level stops explaining reality.

## Write the Handoff Doc

Create or update a short running document when the session is long or the issue is layered.

Include only:

- current source of truth
- confirmed findings
- rejected findings that were previously believed
- fixes already landed
- tests already added or updated
- current blockers
- next highest-value step

Do not turn the handoff into a changelog. Keep it decision-oriented.

## Use External Inspection Deliberately

When local tests pass but the live system still fails, use an external inspector only for focused comparisons. Ask for:

- exact original-vs-converted difference
- exact selector, payload field, event id, or computed style
- first stage where the data becomes wrong

Do not ask for broad exploratory audits when a narrower prompt can isolate the failure.

## End Each Cycle

Before reporting progress, state:

- what is now proven
- what remains broken
- whether the latest result points to converter logic, generated output, or runtime interpretation
